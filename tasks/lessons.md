# Lessons

## 2026-03-09

- 사용자가 "게임같이"를 요구하면 텍스트 UI나 카드형 웹 화면이 아니라 실제 전장 감각을 우선 기준으로 잡을 것
- Linux 실행 환경에서는 한국어 시스템 폰트가 없을 수 있으니 폰트를 저장소에 번들하는 쪽을 먼저 고려할 것
- 비주얼 피드백 요청에서는 로직 구현보다 스크린샷 검증 경로를 먼저 준비할 것
- 사용자 요청에 따라 작업 단위가 생길 때마다 커밋과 푸시를 바로 수행할 것
- 적 대상 스킬의 유효 타겟 조회는 실제 시전뿐 아니라 미리보기와 클릭 하이라이트에서도 같은 결과를 반환해야 함
- 플레이 가능한 전투를 만들었으면 효과음이 없더라도 "완성"으로 보지 말고 기본 오디오 피드백을 함께 넣을 것
- WSL/Linux 배포에서는 Python 패키지만으로 끝나지 않고 오디오 런타임 라이브러리 누락 여부도 같이 확인할 것
- 사용자가 "게임 같다"는 감각을 원하면 전투 화면뿐 아니라 시작 화면, 선택 흐름, 재진입 흐름까지 같이 설계할 것
- 챔피언 선택 풀은 금방 늘어나므로 카드 레이아웃을 고정 수치로 두지 말고 풀 개수에 따라 열과 높이를 계산하도록 만들 것
- `PyInstaller`로 패키징할 때 패키지 내부 `__main__.py`를 직접 진입점으로 쓰면 상대 import가 깨질 수 있으니, 루트 런처 파일을 따로 두는 편이 안전함
- Riot 공식 게임 음원을 바로 쓰기보다, 테마를 닮은 오리지널 사운드를 합성해서 넣는 쪽이 법적 리스크와 유지보수 면에서 낫다
- Windows `.exe`는 Linux/WSL에서 억지로 교차 빌드하지 말고, Windows 러너나 실제 Windows 환경에서 PyInstaller를 돌리는 경로를 기본값으로 둘 것
- 챔피언을 늘릴 때는 아이콘, 전투 데이터, 선택 화면, 사운드 프로필, 테스트를 같이 묶어 업데이트해야 회귀가 적다
- 게임 규칙이 크게 바뀌는 실험은 기존 플레이 가능 빌드를 덮어쓰지 말고, 같은 저장소 안 별도 패키지로 병행하는 편이 안전하다
- `native_tactics`처럼 선택/배치/전투로 상태가 분리된 흐름은 첫 화면만 도는 스모크 테스트로 충분하지 않으니, 최소 한 번은 실제 상태 전환까지 검증할 것
- 종료 화면이 있는 게임 흐름에서는 숨은 단축키만 남기지 말고, 다음 행동을 고르는 버튼을 화면에 직접 제공할 것
- 사용자가 "다음에 뭘 해야 하는지 모르겠다"고 느끼면, 새 콘텐츠 추가보다 먼저 전투 후 목표 제시와 재진입 루프를 설계할 것
- 전술 정보는 사이드 패널 문구만으로 끝내지 말고, 그리드 위 하이라이트와 같이 보여 줘야 이해 속도가 올라간다
- 지형과 엘리트 같은 새 규칙은 전장 색/아이콘/수치 변화/테스트를 한 번에 넣어야 유저도 이해하고 개발도 안정적이다
- 전투 후 선택이 2단계 이상이면 한 화면에 몰아넣기보다 `보상`과 `경로`를 분리한 상태 화면으로 나누는 편이 훨씬 명확하다
- 3안 경로 카드처럼 선택지가 늘어나는 화면은 기능 추가 직후 헤드리스 스크린샷으로 버튼 겹침과 텍스트 밀도를 바로 확인할 것
- 전투 중 계속 보여야 하는 목표 정보는 별도 패널보다 하단 HUD 한 줄 요약이 공간 대비 효율이 높다
- 앱이 들고 있는 전술 목표 상태를 엔진 AI가 참조해야 하면, 생성 시 주입만 믿지 말고 전투 중 상태 변화마다 목표 타일을 다시 동기화할 것
- 경로 카드에 새로운 효과를 넣을 때는 선택 직후 설명 문구를 길게 늘리기보다, 카드 자체와 다음 전투 정보 박스에 요약을 분산하는 편이 가독성이 좋다
- 챔피언 개성이 중요한 전술 모드에서는 `특수기`를 자동 휴리스틱으로 고르지 말고 챔피언별 명시 매핑으로 고정해야 예측 가능성과 테스트 안정성이 유지된다
- 결전 연출을 만들 때는 모든 강적을 `엘리트`로만 뭉개지 말고, `보스 1명 + 엘리트 부관`으로 나눠 배지와 규칙을 구분해야 화면과 전술 양쪽에서 읽히기 쉽다
